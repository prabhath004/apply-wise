# Technical PRD: Local Job Intelligence Chrome Extension

## 1. Product Summary

Build a local-first Chrome extension that helps job seekers analyze a job posting, compare it against their resume, research the company, find likely recruiter contacts, and generate personalized outreach.

The first version must run on the developer's laptop with minimal setup. It should not require a hosted backend, paid database, or production infrastructure.

The product should feel professional, fast, and trustworthy. It should avoid spammy behavior, fake data, or overclaiming. It should clearly show confidence levels when emails or recruiter contacts are inferred.

## 2. Product Name

Working name: **ApplyIntel**

Alternative names:
- RecruitIQ
- JobSignal
- FitReach
- ApplyWise

For the codebase, use neutral naming:

```txt
job-intel-extension
```

## 3. Target User

Primary user:
- New grad software engineers
- CS students
- International students on OPT/F1
- Early-career technical job seekers

Core user problem:
- Users apply to many jobs without knowing if they are a good fit.
- Users do not know which recruiter or hiring contact to reach out to.
- Users waste time on companies that may not be visa-friendly or technically relevant.
- Users send generic outreach messages with low response rates.

## 4. Core Value Proposition

The extension should answer these questions from a job posting page:

1. Am I a good fit for this job?
2. Why am I a good or weak fit?
3. What does this company do?
4. Who are the best people to contact?
5. What email/contact information can be found or inferred?
6. What personalized message should I send?

Positioning:

```txt
Analyze the job. Understand the company. Find the right contact. Apply smarter.
```

## 5. MVP Scope

### 5.1 Must Have

V1 must support:

- Chrome extension using Manifest V3
- LinkedIn job post capture
- Manual fallback input for job description
- Resume upload and local profile extraction
- Bring-your-own OpenAI API key
- Job fit score
- Fit score explanation
- Company summary
- Recruiter/contact discovery using public web search results
- Likely email generation based on public email pattern evidence
- Confidence scoring for contacts/emails
- Outreach message generation
- Local SQLite storage
- Professional sidebar UI
- FastAPI local backend

### 5.2 Nice to Have

V1 may support if simple:

- Save analyzed jobs locally
- Export analysis as Markdown
- Copy outreach message
- Light/dark mode
- Basic job tracker statuses: Saved, Applied, Reached Out, Interview, Rejected

### 5.3 Out of Scope for V1

Do not build in v1:

- Auto-applying to jobs
- Auto-sending emails or LinkedIn messages
- Aggressive scraping
- Paid enrichment APIs
- Hosted accounts/authentication
- Team collaboration
- Browser support beyond Chrome/Chromium
- Mobile support
- Production queue system
- Full SaaS dashboard
- Phone number collection

## 6. Non-Negotiable Product Rules

1. Do not fabricate recruiter names, emails, company facts, funding data, or sponsorship data.
2. If data is inferred, label it clearly as inferred.
3. If email is guessed from a pattern, show confidence and reasoning.
4. Do not scrape private or authenticated data beyond the job page the user is viewing.
5. Do not automate spam or mass outreach.
6. Do not store the user's OpenAI key in the backend database by default.
7. All local data should remain on the user's laptop.
8. Code must be clean, typed, modular, and readable.
9. No emojis in UI, logs, code comments, prompts, or generated content unless the user explicitly asks.
10. No fake placeholder results in production paths.

## 10. User Experience Requirements

### 10.1 UI Design Goal

The extension must look like a professional productivity tool, not a hackathon demo.

Design principles:

- Clean layout
- Neutral colors
- Strong spacing
- Clear hierarchy
- No clutter
- No emojis
- Minimal animations
- Fast perceived performance
- High trust language
- Confidence labels for uncertain data

### 10.2 Sidebar Layout

The main UI should be a right-side sidebar injected into the job page.

Suggested width:

```txt
380px to 440px
```

Sections:

```txt
Header
  - Product name
  - Connection status
  - Settings icon

Job Snapshot
  - Job title
  - Company
  - Location
  - Source page

Fit Score
  - Overall score
  - Recommendation: Strong Apply / Apply / Maybe / Skip

  - Generate message

  - Regenerate
```

### 10.3 Visual Style

Use a restrained SaaS style:

- Background: off-white or very light gray
- Cards: white with subtle border
- Border radius: 12px
- Font: system font stack
- Primary accent: blue or indigo
- Avoid bright gradients
- Avoid noisy shadows
- Use compact status badges

### 10.4 Component States

Every major component must support:

- idle
- loading
- success
- empty
- error
- partial data

Example empty state:


Example error state:


## 11. Extension Behavior

### 11.1 LinkedIn Job Extraction

The extension should detect LinkedIn job pages and attempt to extract:

- job_title
- company_name
- location
- job_description
- job_url
- employment_type if visible
- seniority if visible
- posted_date if visible

The extractor must be defensive because LinkedIn DOM changes often.

Extraction strategy:

1. Try known selectors.
2. Try semantic text fallback.
3. Try structured text sections.
4. If extraction fails, show manual paste form.

Do not crash if selectors fail.

### 11.2 Manual Input Fallback

The user must be able to paste:

- company name
- job title
- job description
- job URL

This is critical because LinkedIn pages may change or block extraction.

### 11.3 Extension to Backend Communication

Default backend URL:

```txt
http://localhost:8000
```

The extension should allow this to be changed in settings.

## 16. Fit Scoring Algorithm

Use a hybrid deterministic + LLM-assisted score.

### 16.1 Score Weights

```txt
Skills match: 40%
Experience relevance: 25%
Seniority fit: 15%
Education fit: 10%
Domain/project fit: 10%
```

### 16.2 Skills Match

Calculate based on:

- exact normalized skill overlap
- synonym mapping
- semantic similarity

Examples:

```txt
React.js = React
PostgreSQL = Postgres
Amazon Web Services = AWS
RESTful APIs = REST APIs
```

### 16.3 Recommendation Thresholds

```txt
85-100: Strong Apply
70-84: Apply
55-69: Maybe
0-54: Low Fit
```

### 16.4 Explanation Rules

The explanation must include:

- top matching skills
- missing required skills
- relevant resume evidence
- risk factors
- clear recommendation

Do not say the user is guaranteed to get the job.

## 17. Company Intelligence Pipeline

Input:

- company name
- job URL
- optional company URL

Output:

- short company summary
- industry
- website
- careers URL
- hiring notes
- source links

V1 strategy:

1. Try to find company website from job page text or user input.
2. Fetch homepage if available.
3. Fetch careers/about pages if discoverable.
4. Summarize only from fetched public pages.
5. If public pages are unavailable, return partial result.

Do not invent:

- funding
- revenue
- layoffs
- sponsorship history
- employee count

Unless a credible source is available.

## 18. Contact Discovery Pipeline

Goal: find likely recruiter or hiring contacts.

Priority order:

1. Technical Recruiter
2. Engineering Recruiter
3. Talent Acquisition Partner
4. University Recruiter
5. Hiring Manager
6. Engineering Manager

V1 should not aggressively scrape LinkedIn.

Acceptable sources:

- company careers page
- company team page
- public personal pages
- public search result snippets if configured
- manually supplied URLs

Contact schema:

```json
{
  "name": "string",
  "title": "string|null",
  "profile_url": "string|null",
  "email": "string|null",
  "email_type": "verified|public|inferred|null",
  "confidence": 0,
  "confidence_reason": "string",
  "sources": []
}
```

## 19. Email Pattern Logic

Email types:

```txt
public: exact email found on a public page
verified: exact email found and clearly tied to the person
inferred: guessed from a company pattern
unknown: no email found
```

Pattern examples:

```txt
first.last@company.com
first@company.com
first_initial_last@company.com
firstlast@company.com
```

Confidence examples:

```txt
95: exact public email tied to person
85: exact email found on company domain, person match likely
70: inferred from multiple public examples of same domain pattern
50: inferred from one weak pattern
0: no email
```

The UI must label inferred emails clearly.

## 21. BYOK OpenAI Key Design

This is an open-source project. Users must be able to bring their own OpenAI key.

### 21.1 Preferred Local Key Flow

1. User opens extension settings.
2. User enters OpenAI API key.
3. Key is stored in Chrome local storage.
4. For backend requests, extension sends key in request header:

```txt
X-OpenAI-API-Key
```

5. Backend uses the key only for that request.
6. Backend does not persist the key unless user explicitly enables persistence.

### 21.2 Backend .env Fallback

Allow local dev fallback:

```txt
OPENAI_API_KEY=
OPENAI_MODEL=gpt-4.1-mini
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
```

Priority order:

1. Request header key
2. Environment variable key
3. Return clear error

Error:

```json
{
  "error": "missing_openai_api_key",
  "message": "Add your OpenAI API key in extension settings or backend .env."
}
```
